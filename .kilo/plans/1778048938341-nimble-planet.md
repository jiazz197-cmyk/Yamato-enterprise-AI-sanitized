# 目标

按用户要求完成三件事：
1. 说明 FileManager 页面任务流程是否有大型改动（已完成分析结论）。
2. 将报价任务队列并发限制改为 2，并增加按 IP 限制。
3. 重新打开前端任务 WebSocket。

---

## 现状结论（基于只读检查）

### A. FileManager 页面流程变更级别
- 属于**中到大型流程改动**，不只是样式。
- 关键变化：
  - 显式关闭任务 WebSocket：`frontend/apps/chat/src/pages/FileManagerPage.vue` 中 `ENABLE_TASK_WEBSOCKET = false`。
  - 任务刷新逻辑重构（事件补丁、本地排队刷新、单任务延迟刷新）。
  - 列表加载引入 `AbortController` 超时和 silent 模式；任务列表 limit 从 300 调整到 100。
  - 全局轮询从 20s 改为 60s。
  - 结果展示改为 compact 摘要优先，降低大结果卡顿风险。

### B. 后端队列限制现状
- 当前是**按 owner_id 的并发限制**，上限 1：
  - `app/core/quotation_dispatcher.py` 初始化 `max_running_per_user=1`。
- 当前并没有“按 IP 调度并发限制”的持久化字段与查询路径。

### C. WebSocket 现状
- 后端 WebSocket 端点和连接管理存在且启用。
- 前端 FileManager 通过开关关闭了任务 ws 连接，导致仅靠轮询兜底。

## 实施计划

## 1. 任务调度并发从 1 改为 2

### 改动点
- `app/core/quotation_dispatcher.py`
  - 将 dispatcher 初始化值从 `max_running_per_user=1` 改为 `2`。

### 预期结果
- 同一限制维度下（当前代码是 owner 维度），可同时运行 2 个任务。

---

## 2. 新增“按 IP 限制”到报价任务调度

> 目标不是 WebSocket 连接限流，而是**报价任务执行并发调度限流**。

### 2.1 数据模型与迁移
- 在 `quotation_tasks` 增加 `owner_ip` 字段（可索引）。
- 补数据库迁移脚本（遵循项目现有迁移体系）。

### 2.2 任务创建链路透传 owner_ip
- `app/api/v1/quotation_generation.py`
  - 在创建任务接口中读取请求来源 IP。
  - 需要兼容代理场景（优先可信代理头；回退 `request.client.host`）。
- `app/usecases/quotation/create_task.py`
  - `CreateQuotationTaskCommand` 增加 `owner_ip`。
- `app/ports/domains/quotation.py`
  - `create_task` port 签名增加 `owner_ip`。
- `app/adapters/quotation/persistence.py`
  - ORM 持久化 `owner_ip`。
- `app/models/orm/quotation_task.py`
  - ORM 字段定义同步更新。

### 2.3 调度器从 owner 维度扩展到 IP 维度
- `app/core/quotation_dispatcher.py`
  - 将“运行中计数 + 出队筛选”增加 IP 维度约束。
  - 方案：
    - 保留 owner 级公平性（按 owner 调度入口不变）。
    - 同时在实际出队前检查 owner_ip 对应运行中任务数 < IP 限制。
  - 将默认并发上限配置化（建议新增 settings 变量）：
    - `QUOTATION_MAX_RUNNING_PER_OWNER=2`
    - `QUOTATION_MAX_RUNNING_PER_IP=2`（或由用户指定）

### 2.4 启动恢复与回调路径
- 检查 `main.py` 的启动恢复逻辑（遍历 owner_id 调度）与 worker done callback（按 owner_id 继续调度）。
- 保持入口兼容（仍可按 owner 触发），但实际出队阶段执行 IP 限制判定，确保全局生效。

## 3. 重新打开 FileManager 的任务 WebSocket

### 改动点
- `frontend/apps/chat/src/pages/FileManagerPage.vue`
  - 将 `ENABLE_TASK_WEBSOCKET = false` 改为 `true`。

### 保留策略
- 保留现有轮询兜底（60s）和超时中断机制，避免 ws 异常时页面完全失去更新。

### 联调检查
- 核对 ws URL 拼接：
  - 前端 `frontend/apps/chat/src/services/ws.ts` 生成 `/document-tasks/ws/{task_id}?token=...`
  - 后端端点 `app/api/v1/websocket_notifier.py` 为 `/ws/{task_id}`，需确保路由前缀组合后路径一致。

## 4. 验证计划

### 后端验证
1. 同一 owner 连续提交 3 个任务：前 2 个应 running，第 3 个 queued。
2. 同一 IP 下不同 owner 提交任务：总 running 数不超过 IP 上限。
3. 任务完成/取消后，队列能自动补位。

### 前端验证
1. FileManager 打开后 `wsConnections` > 0（有活跃任务时）。
2. 任务进度事件可实时更新（不依赖 60s 轮询）。
3. ws 异常断开后仍可通过轮询恢复状态。

### 回归验证
- 审核态（`awaiting_approval`）详情补拉逻辑不受影响。
- 大结果 compact 展示与下载入口正常。

---

## 5. 风险与控制

- 风险：代理环境错误取 IP（全部用户落到网关 IP）。
  - 控制：复用项目已有“可信代理 + X-Forwarded-For”判定思路。
- 风险：历史任务缺少 owner_ip。
  - 控制：迁移时允许 null；调度对空 IP 采用保守策略（仅 owner 限制）并记录告警日志。
- 风险：并发提升后下游资源压力上升。
  - 控制：将 owner/IP 上限配置化，并提供回退到 1 的环境变量。
